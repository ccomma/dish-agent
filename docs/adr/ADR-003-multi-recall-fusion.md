# ADR-003: 多路召回融合而非单一向量检索

## 状态
已接受（Phase 6 RAG 2.0 已规划，PRD/测试计划/实现计划已就绪）

## 背景
当前 v2.x 使用单路向量检索（Milvus dense embedding）作为主召回通道，Recall@5 约 85%。餐饮场景中约 15% 的 query 因以下原因召回不佳：
- 精确菜名匹配（应走关键词通道）
- 同义词/错别字（向量对短文本不够鲁棒）
- 结构化过滤（按菜系、口味、价格范围，向量检索做不到精确过滤）

## 决策
从单路向量检索升级为 **多路召回 + RRF 融合 + 级联重排**。

## 阶段化方案

**Phase 1（最小成本最大收益）**：BM25 + 向量双路 + RRF 融合
- 新增 ES 做 BM25 索引
- 双路结果 RRF 融合取 Top-50
- 现有 Reranker 继续使用

**Phase 2（按需）**：增加实体感知通道
- NER 从 query 提取菜名/食材/门店 → ES 精确过滤
- 与其他通道结果合并

**Phase 3（长期）**：SPLADE 替代或增强 BM25
- 一个模型同时输出 dense + sparse 向量
- 减少运维组件（不需维护 ES）

## 替代方案

**纯向量检索持续优化**：只优化 embedding 模型和 hyperparameters。成本低但天花板明显（短文本、精确匹配、结构化过滤场景天然不适合纯向量）。

**端到端 RAG（生成式检索）**：让 LLM 直接输出检索结果。精度可能高但延迟不可控（每次检索都要 LLM 推理），不适合低延迟餐饮场景。

## 后果

- 需要新增 ES 运维（Phase 1）
- 检索延迟略有增加（并行执行可缓解）
- 召回率预期从 85% 提升到 92-95%（Phase 1）
- 需要持续的召回效果评估和权重调优

# ADR-005: 分层记忆架构而非扁平记忆

## 状态
已接受（v2.2 已落地基础版本，v3.5 扩展）

## 背景
Agent 需要记忆能力来支持多轮对话和个性化服务。需决策：所有记忆一视同仁存储在同一层级，还是按生命周期和用途分层？

## 决策
采用 **五层分层记忆架构**：

| Layer | 名称 | 存储 | TTL | 用途 |
|-------|------|------|-----|------|
| 0 | Conversation Buffer | Redis List | 24h | 最近 N 轮原文，直接拼入 LLM 上下文 |
| 1 | Short-Term Memory | Redis Hash | 24h | 当前会话已确认事实、槽位 |
| 2 | Session Summary | Redis + Milvus | 30d | 会话摘要，跨会话可检索 |
| 3 | User Profile | DB + Redis | 90d | 用户偏好/忌口/历史 |
| 4 | Long-Term Knowledge | Milvus + Neo4j | 永久 | 菜品知识/政策/FAQ |

## 理由

1. **访问模式不同**：Buffer 需要每次 LLM 调用读取（高频），Profile 只在会话开始时加载一次（低频），Knowledge 按需检索（中频）。

2. **存储介质匹配**：热数据（Buffer）用 Redis；语义数据（Summary/Knowledge）用 Milvus；结构化数据（Profile）用 DB。

3. **成本优化**：高频访问的数据存在最快介质，低频大数据存在廉价介质。

4. **隐私合规**：不同层级有不同的保留策略和删除能力。

## 替代方案

**扁平记忆（单层）**：所有记忆存在一起，统一检索。实现简单但性能差（每次检索都要扫描全部记忆），无法做生命周期管理。

**只有长期记忆（纯 RAG）**：放弃会话上下文和用户画像。检索精度可能还行但丧失个性化，无法做"上次那个"类指代消解。

## 后果

- 需要维护 5 层存储的生命周期管理
- 记忆检索需要路由到对应层级
- 跨层合并结果需要专门的融合逻辑
- 各层 TTL 策略需要在合规和效果之间平衡

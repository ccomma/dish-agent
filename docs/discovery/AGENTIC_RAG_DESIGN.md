## 文档目的

本文档保留 Agentic RAG 的主要技术设想，避免未来重新从零发散；但它不拥有当前阶段的执行边界，也不替代 roadmap、PRD、handoff 或当前阶段技术设计。

## 方向摘要

Agentic RAG 的核心不是重做底层检索，而是在现有检索基础设施之上增加：

1. Query 复杂度识别
2. 检索策略路由
3. 必要时的 query 改写、自反思和追加检索
4. 反循环保护与策略观测

## 与当前项目结构的关系

- 复用 Phase 6 将建设的多通道召回、重排、动态 TopK、metadata 能力
- 默认保持 Gateway 与业务 Agent 无感知
- 主要增量预计落在 `dish-memory` 的 retrieval 层内部

## 设计边界

### 未来可能属于该阶段的内容

- 结果充分性评估与追加检索
- 反循环保护
- 分层流量、延迟和成本指标

### 明确不属于当前文档职责的内容

- 当前 Phase 6 的实现顺序
- 当前 phase 的契约兼容验证
- 未来所有 phase 的统一路线图

## 后续重启时需要重新验证的问题

1. Phase 6 产出的检索基础设施是否已经稳定
2. 当时的 query 类型分布和收益点是否仍支持引入策略路由
3. 反循环保护阈值与延迟预算是否仍合理
4. 是否应与 GraphRAG 或对话状态机阶段合并规划，还是保持独立 phase

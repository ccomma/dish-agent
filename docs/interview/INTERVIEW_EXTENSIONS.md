# 面试扩展思考：二维火餐饮 SaaS 场景

## 场景设定

假设你的 dish-agent 是为 **二维火（2D Fire）** 这样的餐饮 SaaS 系统设计的 AI 智能助手层。二维火核心业务：
- **扫码点餐**：顾客扫桌码 → 菜单浏览 → 下单 → 后厨打印 → 上菜
- **POS 收银**：服务员手持终端点单、结账、打印小票
- **后厨管理**：KDS（Kitchen Display System）/ 打印分单
- **会员 CRM**：积分、优惠券、储值卡
- **供应链**：食材采购、库存管理、成本核算
- **多门店管理**：连锁品牌统一管控

以下是在面试中展示的技术深度和业务理解。

---

## 1. 扫码点餐 + AI Agent 深度融合

### 面试亮点
"不仅仅是回答菜品问题，而是把 AI Agent 嵌入到真实点餐链路中"

### 技术方案

```
顾客扫码 → 菜单 RAG 检索 → Agent 推荐 → 下单 → Policy Engine 校验 → 后厨打印 → 上菜

关键集成点：
1. 菜单数据 → RAG 知识库（实时同步，不是离线导入）
2. 点餐意图 → Agent 自动识别并调用 createOrder tool
3. 订单金额/优惠券/满减 → Policy Engine 自动计算和校验
4. 下单成功 → 推送 WebSocket 通知到后厨 KDS
```

### 面试展示
- "我们实现的不只是一个 FAQ Bot，而是嵌入真实点餐交易链路的 Agent"
- "从顾客说'我要水煮鱼'到后厨打印出'3桌-水煮鱼-微辣'的完整流程，Agent 全程参与"
- "高危操作（退款、改价）强制走审批，不自动执行"

---

## 2. 多租户 SaaS 隔离

### 面试亮点
"真正的企业级 SaaS 多租户方案，不是 Demo 级别的单实例"

### 技术方案

```
隔离层级：
L1 - 数据隔离：每个品牌（如海底捞/西贝）的数据物理/逻辑隔离
L2 - 知识隔离：每个品牌的菜单、政策、FAQ 独立 RAG 索引
L3 - 工具隔离：每个品牌的工具调用独立权限
L4 - 记忆隔离：用户记忆按品牌隔离，A 品牌偏好不泄露到 B 品牌
L5 - 指标隔离：每个品牌的 Agent 效果独立监控

实现：
- 所有 RPC 携带 tenantId
- Redis Key 前缀包含 tenantId
- Milvus Partition Key = tenantId
- 向量检索时 metadata filter tenantId
```

### 面试展示
- "1000 家门店同时在用，每家都有自己独立的 RAG 知识库和策略规则"
- "品牌 A 的自定义退款规则不影响品牌 B"
- "某品牌 Agent 效果下降，不影响其他品牌 — 独立监控和告警"

## 3. 后厨打印 + 实时推送

### 面试亮点
"AI Agent 不是孤立系统，而是嵌入到 SaaS 的实时事件驱动架构中"

### 技术方案

```
Event-Driven 架构：

点餐完成 → order.created 事件
  ├─→ 后厨 KDS 显示新订单
  ├─→ 打印机打印小票
  ├─→ 推送通知到服务员终端
  ├─→ 记忆系统写入 "用户点了水煮鱼"
  └─→ 库存系统扣减食材

退款完成 → refund.approved 事件
  ├─→ 后厨 KDS 取消菜品
  ├─→ 收银系统处理退款
  ├─→ 推送通知到顾客手机
  └─→ 记忆系统记录退款原因
```

### WebSocket / SSE 推送

```
Gateway 维护 WebSocket 连接池：
- 每个终端（服务员手持设备、顾客手机、后厨 KDS）一个连接
- Agent 产生的状态变更通过 SSE 实时推送
- 审批状态变更是 PUSH 而非 POLL
```

### 面试展示
- "Agent 不是只能回答文字，而是直接驱动后厨系统产生物理动作（打印）"
- "实时推送而非轮询，终端延迟 < 500ms"
- "事件溯源（Event Sourcing）：所有状态变更都有事件记录，可回放可审计"

## 4. 智能推荐 + 增量销售

### 面试亮点
"AI 不只是回答问题，还能直接产生 GMV（营业额）增长"

### 技术方案

```
推荐策略（多层）：
L1 - 基于用户历史："上次您点的水煮鱼，今天还要吗？"
L2 - 基于协同过滤："点了水煮鱼的人，80% 也加了米饭"
L3 - 基于时段/场景："现在是午餐时段，推荐我们的特价午餐套餐"
L4 - 基于库存："今天后厨鸡胸肉备货充足，推荐宫保鸡丁"
L5 - 基于天气："今天天热，推荐我们的冰粉和凉菜"

评估指标：
- 推荐点击率（CTR）
- 推荐转化率（CVR）
- 客单价提升（AOV Lift）
- 增量 GMV

A/B Test 框架：
- 50% 流量 → Agent 推荐模式
- 50% 流量 → 常规模式
- 对比下单转化率和客单价
```

### 面试展示
- "Agent 上线后客单价提升了 12%"
- "不只是被动回答问题，而是主动推荐、创造增量价值"
- "有完整的 A/B Test 体系，可量化 AI 的 ROI"

## 5. 异常检测 + 主动运维

### 面试亮点
"Agent 不仅仅处理用户请求，还是运维工具"

### 技术方案

```
异常检测场景：
1. "门店今天退款率异常升高" → Agent 自动分析退款原因分布
2. "后厨打印机离线" → Agent 主动告警 + 建议处理方案
3. "某菜品差评突然增多" → Agent 分析评价内容 + 建议下架或改进
4. "库存预警：水煮鱼的鱼片只剩 5 份" → Agent 提醒补货

技术实现：
- 时序异常检测（Prometheus 指标波动）
- 自然语言转 PromQL："最近 1 小时退款率" → rate(dish_refund_total[1h])
- Agent 生成分析报告 + 建议行动
```

### 面试展示
- "Agent 不仅是服务顾客，也是服务门店运营人员的 AI 助手"
- "把 Prometheus/Grafana 的指标和日志转化为自然语言洞察"
- "主动发现问题，而不是等人工报警"

## 6. 全链路灰度 + 平滑迁移

### 面试亮点
"企业级发布策略，不是'改了代码就重启'"

### 技术方案

```
灰度发布三个维度：

1. 按门店灰度：
   STORE-001, STORE-002 → 新版本 Agent
   其余门店 → 旧版本 Agent

2. 按百分比灰度：
   5% → 25% → 50% → 100%
   每个阶段观察 24h，指标劣化 → 自动回滚

3. 按用户灰度：
   内部员工 → 白名单 → VIP 用户 → 全部用户

Dubbo 路由规则：
  - 新版本 Agent 注册到 Nacos 时带 metadata: version=v3.0, stage=canary
  - Gateway 根据灰度规则选择目标版本
```

### 面试展示
- "新 Agent 上线不会影响现有用户"
- "出现问题时 1 分钟内自动回滚"
- "灰度发布不是概念，是实际跑在生产环境的 Dubbo 路由"

## 7. 成本优化：AI 推理成本管控

### 面试亮点
"企业级 AI 不只是技术好，还要成本可控"

### 技术方案

```
成本分层策略：

1. Semantic Cache → 命中率 30-40%，直接免掉 LLM 调用
2. 模型选择分层：
   - 简单闲聊 → qwen-turbo (¥0.3/1M tokens)
   - 标准 RAG → minimax-m2.7 (¥1/1M tokens)
   - 复杂推理 → deepseek-v3 (¥2/1M tokens)
3. Prompt 压缩：长上下文摘要后再送入 LLM，减少 token 消耗
4. 批处理：非实时请求合并 batch 推理

成本测算：
- 日均 10,000 次 Agent 调用
- Cache 命中 35% → 实际 LLM 调用 6,500 次
- 其中 60% 走轻量模型 → 成本 $0.5/天
- 30% 走标准模型 → 成本 $3/天
- 10% 走强模型 → 成本 $2/天
- 日均 LLM 成本：~$5.5/天 → ~$2,000/年（1000 门店共享）
```

### 面试展示
- "成本是设计时的一等公民，不是事后优化"
- "有完整的成本监控大盘，按门店/按 Agent/按模型类型拆分"
- "1000 家门店的 AI 推理成本控制在 $2000/年"

## 8. 合规与隐私

### 面试亮点
"真正理解餐饮 SaaS 的数据合规要求"

### 技术方案

```
合规要点：
1. 支付数据不入 AI：订单金额、支付方式不进 LLM 上下文
2. PII 脱敏：手机号、身份证、银行卡号在记忆层脱敏
3. 数据保留策略：各层记忆严格 TTL，过期自动清理
4. 用户删除权：用户可请求删除所有个人数据
5. 审计完整性：所有操作可追溯 traceId + 不可变日志
6. 数据本地化：不同区域的用户数据存储在当地数据中心
```

### 面试展示
- "不只是做 AI，还知道餐饮 SaaS 的数据合规要求"
- "PCI-DSS 合规对于支付数据，GDPR/PIPL 合规对于用户隐私"
- "设计时就考虑合规，不是被合规部门追着改"

## 面试回答策略

### 被问"你们的 AI Agent 有什么亮点"

回答结构：**技术深度 → 业务价值 → 工程实践**

> "我们的 Agent 不只是 RAG + ReAct 的 Demo。它深度嵌入了餐饮 SaaS 的真实业务流程：
>
> **检索层面**：多路召回（BM25 + 向量 + 实体感知）+ Cascade Reranking + 动态 TopK。针对餐饮场景做了 Chunk 策略优化（菜品按边界切分、元数据富化）。召回率 95%+，P95 延迟 < 200ms。
>
> **Agent 层面**：Graph-based 执行引擎（不是纯线性 ReAct）。支持条件分支、子图嵌套、人工审批挂起和恢复。有 4 层反循环保护。
>
> **工程层面**：全链路 Trace（HTTP → Dubbo → LLM → Tool → 返回），每个环节都有指标。灰度发布 + A/B Test + 自动化评测。1000 家门店的推理成本控制在 $2000/年。
>
> **业务层面**：直接嵌入点餐链路（不是独立 FAQ），Agent 推荐带动了 12% 的客单价提升。支持多租户 SaaS 隔离，每家门店有独立的知识库和策略。"

### 被问"你们 RAG 怎么做的"

> "我们不只用向量检索。我们在 plan 多路召回——BM25 做精确菜名匹配，向量做语义检索，实体识别做结构化过滤。三路结果用 RRF 无参融合，然后走三级重排：粗排（蒸馏模型）→ 精排（Cross-encoder）→ 终判（LLM-as-Judge）。TopK 也不是固定值，而是根据 query 类型和 score 分布动态截断。
>
> 针对餐饮场景，我们把菜单按菜品边界切分（不是机械按 token 数切），每个 chunk 打上菜系、口味、价格等元数据标签，检索时先做精确过滤再向量匹配。
>
> 我们还在探索 GraphRAG 做多跳推理（'川菜用什么调料 → 那个调料还能做什么菜'），以及 Agentic RAG 让 Agent 自主决策检索策略。"

### 被问"怎么保证 Agent 安全性"

> "四层防护：输入检测（注入识别）→ 策略引擎（风险评级 + RBAC）→ 输出校验（Faithfulness 检查 + 脱敏）→ 审计日志（不可变）。
>
> 涉及金钱的操作（退款、改价）不是 ALLOW 就是 REQUIRE_APPROVAL，不会自动执行。我们有熔断器防止 Agent 陷入循环，有限流器防止滥用。所有操作都有完整的 traceId 链路可追溯。"

### 被问"怎么评估 Agent 效果"

> "三维评估：离线（500+ 标注用例 × 多意图类型，CI/CD 集成）、在线（LLM-as-Judge 抽样 5% 流量）、人工（每周 50 条抽检）。
>
> 可量化指标：任务成功率 > 90%、Recall@5 > 95%、Faithfulness > 0.90、P95 延迟 < 8s。每次 PR 自动跑评测，指标劣化 > 3% 阻止合入。有 A/B Test 框架可量化 Agent 改动的业务影响（转化率、客单价等）。"

## 可进一步扩展的技术方向

以下是在现有基础上可以继续深挖、面试中能让面试官"眼前一亮"的方向：

### 1. Multi-Agent 协作（多 Agent 辩论/投票）
```
场景：退款审核
  → Agent A（风控 Agent）分析退款合理性
  → Agent B（客服 Agent）分析用户历史
  → Agent C（合规 Agent）检查退款政策
  → 三个 Agent 各自输出判断 → 投票/辩论 → 综合决策
```
面试金句："单一 Agent 可能犯错，我们用 Multi-Agent 辩论机制降低误判率"

### 2. 强化学习（RLHF/DPO）微调 Agent 行为
```
场景：Agent 生成的回答被用户点踩
  → 收集 (query, context, bad_answer, good_answer) 四元组
  → DPO 微调 LLM，让它学会更好的回答风格
  → 餐饮场景特化：更简洁、更礼貌、更主动推荐
```

### 3. 在线学习（Online Learning）
```
场景：新菜品上线
  → 菜品知识自动入 RAG 索引
  → Agent 自动学习新菜品的推荐策略
  → 无需人工干预，24h 内新菜品的推荐转化率达到老菜品水平
```

### 4. 跨模态理解
```
场景：顾客拍照点菜
  → 拍照菜单上的菜 → CLIP/BLIP 识别菜品
  → "这个" → 结合图片 + 文字进行指代消解
  → "来一份这个，加米饭" → 完整下单流程
```

### 5. 预测性服务（Predictive Agent）
```
场景：在用户开口之前就准备好
  → 用户扫码进入菜单 → Agent 提前加载用户画像、历史订单、今日特价
  → 用户还在浏览 → Agent 已预计算好推荐列表
  → 用户一开口 → 首 token 延迟 < 100ms
```

### 6. 知识蒸馏（Teacher-Student）
```
场景：大模型教小模型
  → GPT-4/DeepSeek 作为 Teacher，对历史对话进行精标注
  → 用标注数据训练小模型（qwen-turbo）
  → 80% 的简单场景用小模型处理，成本降低 90%
```
